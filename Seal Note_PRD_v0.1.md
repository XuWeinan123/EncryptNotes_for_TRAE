# Seal NotePRD v0.1

## 1. 产品范围

Seal Notev0.1 只面向 Apple 平台，首发只承诺 iPhone 体验。

本版本不做账号系统，不做自建服务器，不做 Android，不做 Web，不做多人协作，不做 flomo 批量导入，不做订阅制。

v0.1 的目标是完成一个最小但完整的加密卡片笔记闭环：

> 用户可以在 iPhone 上创建卡片笔记；笔记内容在本地加密后，以多个加密 JSON 文件的形式保存到 iCloud；在未导入密钥文件时，用户只能看到加密后的乱码内容；导入正确密钥文件后，App 在本机将这些文件解密并展示为可读卡片。

---

## 2. 核心产品逻辑

产品不依赖平台账号来同步数据，而是借助 Apple ID 的 iCloud 文件同步能力。

核心流程：

```text
用户创建笔记
→ App 在本地用密钥加密笔记内容
→ 生成一个可同步的加密 JSON 文件
→ 文件保存到 iCloud 容器
→ 其他 Apple 设备通过同一 Apple ID 同步这些文件
→ 未导入密钥时，文件内容显示为加密乱码
→ 导入正确密钥后，本机解密并展示为明文笔记
```

这里的“明文文件”指的是：文件本身可以被普通文件系统读取、复制、移动、备份，但文件内容是加密后的乱码。

如果用户把这些文件复制给别人，对方只有在拥有相同密钥文件的情况下，才能将其解密为可读笔记。

## 4. 数据文件设计

### 4.1 多文件同步

v0.1 不使用单一加密数据库文件，而是采用多个 JSON 文件保存笔记数据。

建议结构：

```text
iCloud Drive / Seal Note /
  vault.json
  notes /
    note_01.bkwenc.json
    note_02.bkwenc.json
    note_03.bkwenc.json
  trash /
  meta /
```

每条笔记对应一个独立加密 JSON 文件。

这样设计的好处：

1. 方便 iCloud 增量同步。
2. 方便后续扩展到 Android、Web 或其他云存储。
3. 方便用户理解：每个文件都是一条被加密的卡片。
4. 方便文件复制、备份和迁移。
5. 更适合展示“未解锁乱码态”。

---

## 5. 文件内容规则

每个笔记文件本身是普通 JSON 文件，可以被复制、移动、备份、分享。

但 JSON 中的笔记内容不是可读笔记，而是加密后的文本。

示例：

```json
{
  "version": 1,
  "app": "BieKanWo",
  "type": "encrypted_note",
  "note_id": "A7F3D9B2-82D1-4A41-9C22-31F9B21A901C",
  "vault_id": "8E0A9B6C-44F3-4B88-8E8A-04B9B6C78111",
  "created_at": "2026-06-18T12:00:00Z",
  "updated_at": "2026-06-18T12:05:00Z",
  "encryption": {
    "algorithm": "AES-GCM",
    "key_version": 1,
    "nonce": "base64_nonce_here"
  },
  "payload": {
    "ciphertext": "U2FsdGVkX1+f9aQ8zR1x7b3a9c11d8f04aa29fA9dKx03LqPz8nV2...",
    "tag": "base64_auth_tag_here"
  }
}
```

### 5.1 可以明文保存的字段

JSON 中可明文保存文件管理所需的最低限度元数据：

- `version`
- `app`
- `type`
- `note_id`
- `vault_id`
- `created_at`
- `updated_at`
- `encryption.algorithm`
- `encryption.key_version`
- `encryption.nonce`

### 5.2 不能明文保存的内容

JSON 中不应明文保存：

- 笔记正文
- 标题
- 标签
- 摘要
- 搜索索引
- 附件内容

## 6. 密钥文件设计

### 6.1 密钥文件定位

密钥文件是打开笔记的唯一凭证。

v0.1 采用手动密钥文件方案。首次创建加密空间时，App 生成密钥文件，并要求用户主动保存。

App 不默认把密钥托管给平台，也不依赖账号恢复密钥。

### 6.2 密钥文件命名

建议默认命名：

```text
Seal Note-密钥-2026-06-18.bkwkey
```

### 6.3 密钥文件规则

用户需要妥善保存密钥文件。

如果用户将加密笔记文件复制给别人，对方只有同时拥有对应密钥文件，才能解密查看。

如果用户丢失密钥文件，App 无法恢复旧笔记。

### 6.4 导入密钥

首页顶部提供明显入口：

```text
导入密钥文件
```

不建议使用“上传密钥文件”作为正式文案，因为“上传”容易让用户误以为密钥会被传到服务器。

辅助说明：

```text
密钥文件只会在本机读取，不会上传。
```

---

## 7. 首页状态设计

### 7.1 未导入密钥状态

当 App 发现 iCloud 中存在加密笔记文件，但当前设备未导入密钥时，首页展示密文卡片流。

顶部提示：

```text
这台设备还没有密钥
导入密钥文件后，笔记将在本机解密显示。
```

主按钮：

```text
导入密钥文件
```

卡片内容展示为加密乱码，例如：

```text
U2FsdGVkX1+f9aQ8zR1x...
7b3a9c11d8f04aa29f...
A9dKx03LqPz8nV2...
```

卡片底部可以展示：

```text
已加密 · 2026.06.18 · 1.4 KB
```

未导入密钥时，用户不允许：

- 创建笔记
- 编辑笔记
- 搜索笔记
- 查看标签
- 查看附件
- 导出明文
- 批量操作明文内容

未导入密钥时，用户允许：

- 查看加密文件列表
- 查看密文卡片
- 导入密钥文件
- 查看密钥丢失说明
- 重置加密空间

---

### 7.2 正在解密状态

用户选择密钥文件后，App 在本机校验密钥并解密笔记文件。

页面展示：

```text
正在本机解析密钥
密钥文件不会上传。
```

随后展示：

```text
正在解密笔记 24 / 128
```

解密过程中，卡片可以逐步从乱码变成可读内容。

---

### 7.3 已导入密钥状态

导入正确密钥后，首页展示明文卡片流。

顶部状态：

```text
已解锁
当前笔记已在本机解密显示。
```

辅助说明：

```text
iCloud 中保存的是加密文件。
```

用户可以：

- 创建笔记
- 编辑笔记
- 删除笔记
- 搜索笔记
- 使用标签
- 锁定 App
- 重新导出密钥文件
- 重置加密空间

---

## 8. 新建笔记规则

未导入密钥时不允许新建笔记。

原因是：没有密钥，App 无法把新笔记加密进当前加密空间。如果此时允许新建，容易导致多个密钥空间混乱。

未解锁状态下，底部输入框显示为禁用状态：

```text
导入密钥后即可记录
```

已导入密钥后，用户可以正常新建笔记：

```text
输入内容
→ 本地加密
→ 生成加密 JSON 文件
→ 写入 iCloud
→ 首页显示明文卡片
```

---

## 9. 重置加密空间

v0.1 需要支持重置加密空间。

重置加密空间的真实含义是：

```text
重新生成密钥
+ 清空 iCloud 中已有的加密笔记文件
+ 创建新的加密空间
```

重置前必须强提醒用户：

```text
重置后，当前 iCloud 中的加密笔记文件将被清空。
如果你没有旧密钥文件，这些笔记将无法恢复。
```

用户确认后执行：

1. 删除 iCloud 中当前加密空间的文件。
2. 清空本地缓存。
3. 生成新的密钥文件。
4. 引导用户保存新密钥文件。
5. 创建新的空加密空间。

重置后，旧密钥不再适用于新加密空间。

## 10. Free / Pro 版本区分

v0.1 只区分 Free 和 Pro。

不做商业版，不做订阅制。

### 10.1 Free

Free 版本只限制笔记数量。

```text
Free：最多 20 条笔记
```

规则：

- Free 用户最多可创建 20 条笔记。
- 达到 20 条后，仍可查看、编辑、删除已有笔记。
- 达到限制后，不允许继续新建笔记，并引导一次性买断 Pro。
- Free 与 Pro 使用完全相同的加密逻辑，不因付费状态削弱隐私保护。

Free 版本仍然支持完整的核心安全机制：

- 本地加密
- 密钥文件生成
- 密钥文件导入
- iCloud 加密 JSON 文件同步
- 未导入密钥时显示乱码
- 导入密钥后本机解密
- App 锁定
- 重置加密空间

### 10.2 Pro

Pro 为买断制。

Pro 解锁：

```text
无限笔记数量
```

v0.1 阶段，Pro 不需要设计过多复杂权益。最核心的付费点就是解除笔记数量限制。

## 12. v0.1 验收标准

### 12.1 加密验收

1. 笔记正文不得以明文形式保存在 iCloud 文件中。
2. 标签不得以明文形式保存在 iCloud 文件中。
3. 未导入密钥时，只能看到加密乱码。
4. 导入错误密钥时，不能解密笔记。
5. 导入正确密钥时，可以在本机解密笔记。
6. 将加密笔记文件复制给别人后，对方没有密钥无法查看内容。
7. 将加密笔记文件和密钥文件一起复制给别人后，对方可以解密查看。
8. App 切后台后，不展示明文内容。
9. App 锁定后，明文内容不可见。
10. 重置加密空间后，旧文件被清空，新密钥被生成。

### 12.2 同步验收

1. 设备 A 创建笔记后，iCloud 中生成对应加密 JSON 文件。
2. 设备 B 使用同一 Apple ID 后，可以看到该加密 JSON 文件。
3. 设备 B 未导入密钥时，只显示乱码。
4. 设备 B 导入正确密钥后，可以解密显示笔记。
5. 删除笔记后，对应 iCloud 文件被删除或写入删除状态。
6. 修改笔记后，对应加密 JSON 文件被更新。
7. 多设备同步冲突时，不应直接覆盖用户内容。

### 12.3 Free / Pro 验收

1. Free 用户最多创建 20 条笔记。
2. Free 用户达到限制后，仍可查看、编辑、删除已有笔记。
3. Free 用户达到限制后，新建入口引导升级 Pro。
4. Pro 用户可创建无限笔记。
5. 升级 Pro 后，原有加密数据不受影响。
6. Free 和 Pro 使用同一套加密逻辑。

### 12.4 重置验收

1. 用户可以从设置中进入重置加密空间。
2. 重置前必须展示高风险确认。
3. 用户确认后，清空 iCloud 中当前加密空间文件。
4. 用户确认后，清空本地缓存。
5. 用户确认后，生成新的密钥文件。
6. 新密钥不能解开旧文件。
7. 旧密钥不能解开新文件。

---

## 13. 当前阶段最终目标

v0.1 的目标不是做一个功能完整的笔记工具，而是先验证一个最关键的产品闭环：

> iPhone 用户可以创建本地加密卡片笔记；这些笔记以 JSON 加密文件同步到 iCloud；没有密钥时只显示乱码；导入密钥后在本机解密显示；Free 最多 20 条，Pro 买断无限。

只要这个闭环跑通，产品就有了继续扩展的基础。
