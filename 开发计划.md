# Seal Notev0.1 开发计划

面向 Trae 的实现计划。依据 `/Users/aaronxu/Downloads/Seal Note_PRD_v0.1.md` 编写，当前仓库是一个几乎空的 SwiftUI iOS 工程，入口在 `EncryptNotes/ContentView.swift`。

## 0. 开发目标

实现一个 iPhone 优先的加密卡片笔记闭环：

```text
创建笔记
-> 本机 AES-GCM 加密
-> 生成单条笔记对应的 .bkwenc.json 文件
-> 写入 iCloud 容器
-> 未导入密钥时只展示密文乱码卡片
-> 导入正确 .bkwkey 密钥文件后本机解密展示明文卡片
-> Free 最多 20 条，Pro 买断无限
```

v0.1 不做账号、自建服务端、Android、Web、Mac 专门适配、富文本、附件、图片、语音、多人协作、服务端搜索、服务端备份、订阅制。

## 1. 关键技术选择

- UI：SwiftUI，iPhone 优先，兼容 iPad 运行但不专门优化。
- 状态管理：`@Observable` 或 `ObservableObject` 的单一 `AppModel`/`VaultStore`，避免页面各自读写文件。
- 加密：Apple CryptoKit `AES.GCM`，256-bit 对称密钥。
- 文件同步：iCloud Documents / ubiquity container，使用 `FileManager.url(forUbiquityContainerIdentifier:)` 获取容器。
- 本地安全存储：导入后的密钥可存入 Keychain；明文笔记只放内存，不写入磁盘。
- 文件导入导出：SwiftUI `fileImporter` / `fileExporter` 或 `UIDocumentPickerViewController` 包装。
- 付费：StoreKit 2，非消耗型内购，一次性解锁 Pro。
- 测试：优先覆盖 Crypto、JSON schema、文件读写、Free 限制和错误密钥场景。

## 3. 数据结构设计

### 3.1 加密空间元数据

文件：`vault.json`

用途：只保存加密空间级别的非敏感元数据，不保存密钥。

建议结构：

```json
{
  "version": 1,
  "app": "Seal Note",
  "type": "vault",
  "vault_id": "UUID",
  "created_at": "ISO-8601",
  "updated_at": "ISO-8601",
  "key_version": 1
}
```

### 3.2 密钥文件

扩展名：`.bkwkey`

建议结构：

```json
{
  "version": 1,
  "app": "Seal Note",
  "type": "vault_key",
  "vault_id": "UUID",
  "key_version": 1,
  "algorithm": "AES-GCM-256",
  "created_at": "ISO-8601",
  "key_material": "base64-32-byte-key"
}
```

实现要求：

- 首次创建加密空间或重置加密空间时生成。
- 引导用户主动保存，不自动上传到服务器。
- 导入时校验 `app`、`type`、`vault_id`、`key_version`、`key_material` 长度。
- 导入后可写入 Keychain，用于当前设备后续解锁。
- 不要将密钥文件复制到 iCloud 笔记目录。

### 3.3 加密笔记文件

目录：`notes/`

文件名：`<note_id>.bkwenc.json`

结构遵循 PRD：

```json
{
  "version": 1,
  "app": "Seal Note",
  "type": "encrypted_note",
  "note_id": "UUID",
  "vault_id": "UUID",
  "created_at": "ISO-8601",
  "updated_at": "ISO-8601",
  "encryption": {
    "algorithm": "AES-GCM",
    "key_version": 1,
    "nonce": "base64_nonce"
  },
  "payload": {
    "ciphertext": "base64_ciphertext",
    "tag": "base64_tag"
  }
}
```

明文笔记内容在加密前统一编码为：

```json
{
  "title": "可为空，默认从正文首行生成",
  "body": "正文",
  "tags": ["tag1", "tag2"],
  "created_at": "ISO-8601",
  "updated_at": "ISO-8601"
}
```

注意：`title`、`body`、`tags` 只能存在于加密 payload 内，不能写入外层 JSON。

### 3.4 删除与冲突

v0.1 建议采用简单可控策略：

- 删除笔记：直接删除对应 `.bkwenc.json` 文件。
- iCloud 删除失败：保留 UI 错误提示，不从内存中假装删除成功。
- 多设备冲突：不要静默覆盖。
  - 保存前比较磁盘文件 `updated_at` 与内存版本。
  - 如果磁盘版本更新，生成冲突副本：`<note_id>-conflict-<timestamp>.bkwenc.json`。
  - UI 用普通卡片展示冲突副本，避免丢内容。

## 4. 核心模块拆分

### 4.1 Crypto

文件建议：

```text
Crypto/VaultKey.swift
Crypto/CryptoService.swift
Crypto/KeychainStore.swift
```

职责：

- 生成 32 字节随机密钥。
- AES-GCM 加密明文 JSON data。
- AES-GCM 解密 encrypted note。
- 错误密钥解密必须失败，不返回任何明文。
- Keychain 读写导入后的密钥。
- App 锁定时清空内存中的 decrypted notes；是否清空内存 key 由实现决定，但 UI 不得展示明文。

验收点：

- 单元测试验证同 key 可解密，错 key 不可解密。
- 单元测试验证密文 JSON 中不包含正文、标题、标签原文。

### 4.2 Storage

文件建议：

```text
Storage/ICloudVaultStorage.swift
Storage/LocalFallbackStorage.swift
Storage/EncryptedNoteFile.swift
Storage/VaultManifest.swift
```

职责：

- 初始化目录：

```text
iCloud Drive / Seal Note /
  vault.json
  notes/
  trash/
  meta/
```

- 读写 `vault.json`。
- 扫描 `notes/*.bkwenc.json`。
- 写入、更新、删除单条 note 文件。
- iCloud 不可用时给出明确 UI 状态；开发阶段可以临时使用本地 fallback，但生产路径必须走 iCloud。
- 监听 iCloud 文件变化，刷新列表。

实现建议：

- 使用 actor 或串行队列保护文件读写。
- 所有写入先写临时文件，再原子替换目标文件。
- 文件大小显示用于未解锁卡片底部信息。

### 4.3 Store / App State

文件建议：

```text
Stores/VaultStore.swift
Stores/PurchaseStore.swift
Stores/AppLockStore.swift
```

`VaultStore` 状态：

```swift
enum VaultState {
    case noVault
    case locked(encryptedFiles: [EncryptedNoteFile])
    case unlocking(progress: UnlockProgress)
    case unlocked(notes: [Note])
    case error(message: String)
}
```

职责：

- App 启动时初始化 iCloud 目录并扫描文件。
- 如果 Keychain 有可用密钥，尝试自动解锁。
- 没有密钥但有加密文件时展示 locked 状态。
- 导入密钥后进入 unlocking，逐条解密并更新进度。
- 新建/编辑/删除笔记时更新内存和 iCloud 文件。
- Free 限制判断由 `VaultStore` 调 `PurchaseStore` 完成，不能只在 UI 层禁用。

## 5. UI 页面计划

### 5.1 首页外壳

文件建议：

```text
Views/HomeView.swift
Views/Components/NoteCardView.swift
Views/Components/EncryptedCardView.swift
Views/Components/BottomComposerView.swift
```

布局：

- 顶部状态区。
- 卡片流列表。
- 底部输入框或禁用输入框。
- 设置入口。

### 5.2 未导入密钥状态

必须展示：

```text
这台设备还没有密钥
导入密钥文件后，笔记将在本机解密显示。
```

主按钮：

```text
导入密钥文件
```

辅助说明：

```text
密钥文件只会在本机读取，不会上传。
```

卡片：

- 用 `payload.ciphertext` 截断成 2-3 行乱码。
- 底部显示 `已加密 · yyyy.MM.dd · <size>`。

限制：

- 不允许创建、编辑、搜索、标签、导出明文。
- 底部输入框禁用，文案：`导入密钥后即可记录`。

### 5.3 正在解密状态

展示：

```text
正在本机解析密钥
密钥文件不会上传。
正在解密笔记 24 / 128
```

实现：

- 逐条解密，进度实时更新。
- 可让已经解密的卡片先显示明文，未解密的继续显示乱码。
- 错误密钥要中断并提示，不要部分进入 unlocked 状态。

### 5.4 已解锁状态

展示：

```text
已解锁
当前笔记已在本机解密显示。
iCloud 中保存的是加密文件。
```

功能：

- 新建笔记。
- 编辑笔记。
- 删除笔记。
- 搜索笔记，搜索只对内存明文执行。
- 标签展示与过滤，标签只来自解密 payload。
- 锁定 App。
- 重新导出密钥文件。
- 重置加密空间。

### 5.5 编辑体验

v0.1 不做富文本，使用纯文本：

- 点击卡片进入编辑页或 sheet。
- 标题可自动取正文首行，用户也可手动编辑。
- 保存时重新加密整个 payload 并覆盖对应文件。
- 空正文不允许保存。
- 保存失败要保留编辑内容并提示。

### 5.6 设置页

文件建议：

```text
Views/SettingsView.swift
Views/ResetVaultView.swift
Views/PaywallView.swift
```

内容：

- 当前状态：已锁定 / 已解锁 / iCloud 不可用。
- 导入密钥文件。
- 导出当前密钥文件，只有已解锁时可用。
- 锁定 App。
- 重置加密空间。
- Pro 状态与升级入口。

重置确认文案必须包含：

```text
重置后，当前 iCloud 中的加密笔记文件将被清空。
如果你没有旧密钥文件，这些笔记将无法恢复。
```

重置流程：

1. 二次确认。
2. 删除当前 vault 下 `notes/`、`trash/`、`meta/`、`vault.json`。
3. 清空本地缓存和 Keychain 旧 key。
4. 生成新 `vault_id` 和新 key。
5. 写入新 `vault.json`。
6. 引导用户保存新 `.bkwkey`。
7. 回到空的已解锁状态。

## 6. Free / Pro 开发计划

### 6.1 Free 限制

- Free 最多创建 20 条笔记。
- 达到 20 条后：
  - 仍可查看、编辑、删除已有笔记。
  - 新建入口打开 Pro 引导。
  - 后台逻辑也必须拒绝创建第 21 条，不能只依赖按钮禁用。

### 6.2 StoreKit 2

文件建议：

```text
Purchase/PurchaseStore.swift
Purchase/StoreProductIDs.swift
```

任务：

- 配置一个非消耗型内购 product id，例如 `pro_lifetime`。
- 加载商品。
- 购买、恢复购买、监听 transaction updates。
- 本地缓存 Pro entitlement，但以 StoreKit transaction 为准。
- 升级 Pro 后不改动任何加密文件。

开发阶段：

- 新增 StoreKit configuration 文件用于模拟 Free/Pro。
- 单元或 UI 测试覆盖 Free 20 条限制。

## 7. App 锁定与后台保护

要求：

- App 进入后台或 inactive 时，立即隐藏明文 UI。
- App switcher 截图中不能出现明文笔记。
- 锁定后首页回到 locked/unlocked-but-hidden 的安全状态，需要用户手动解锁或重新导入密钥。

实现建议：

- 监听 `scenePhase`。
- 进入 background 前将 `VaultStore` 的明文 notes 清空或切换为 hidden。
- 用遮罩 view 覆盖当前界面，避免系统截图捕获明文。
- 如 Keychain 内有 key，回前台可提供“解锁”按钮重新解密。

## 8. 测试计划

### 8.1 单元测试

新增 test target，覆盖：

- `CryptoService`：
  - 生成 key 长度正确。
  - 同 key 加解密成功。
  - 错 key 解密失败。
  - 篡改 ciphertext/tag/nonce 后解密失败。
- `EncryptedNoteFile`：
  - JSON 编解码稳定。
  - 外层 JSON 不包含 title/body/tags 明文。
- `VaultStore`：
  - 未导入密钥时不能创建笔记。
  - Free 第 21 条创建失败。
  - Pro 不限制创建数量。
  - 删除笔记会调用 storage 删除文件。
- `ResetVault`：
  - 清空旧文件。
  - 新旧 key 互相不能解密。

### 8.2 手动验收

按 PRD 验收标准逐项执行：

- 创建一条包含明显关键词的笔记，例如 `我的银行卡密码测试文本`，检查 iCloud JSON 不含该字符串。
- 删除本机 Keychain key 后重启 App，只能看到乱码。
- 导入错误 `.bkwkey`，必须解密失败。
- 导入正确 `.bkwkey`，明文恢复。
- 复制 `.bkwenc.json` 给另一台设备，无 key 只显示乱码。
- 复制 `.bkwenc.json` 和 `.bkwkey` 给另一台设备，可以解密。
- App 切后台，系统多任务界面不展示明文。
- 重置后旧文件清空，新 key 不能解旧文件。
- Free 创建到 20 条后，第 21 条被 Pro 引导拦截。
- Pro 模拟购买后可继续创建。

## 10. Trae 实现注意事项

- 不要把任何明文标题、正文、标签、摘要、搜索索引写入 iCloud 文件。
- 不要创建服务端、账号系统、登录页或云数据库。
- 不要把“导入密钥文件”写成“上传密钥文件”。
- 不要在 locked 状态允许新建笔记。
- 不要用单一数据库文件替代多 `.bkwenc.json` 文件。
- 不要为了搜索方便在外层 JSON 存明文摘要或关键词。
- 不要在重置时只清本地缓存，必须清 iCloud 当前加密空间文件。
- 不要静默覆盖多设备冲突内容。
- 不要让 Free / Pro 使用不同加密逻辑。

## 11. Definition of Done

- iPhone 真机可创建、编辑、删除加密卡片笔记。
- iCloud 中每条笔记是独立 `.bkwenc.json` 文件。
- 未导入密钥时只展示乱码卡片。
- 导入正确 `.bkwkey` 后本机解密展示明文。
- 错误密钥无法解密。
- App 后台和锁定状态不展示明文。
- 重置加密空间会清空旧 iCloud 文件并生成新 key。
- Free 最多 20 条，Pro 买断后无限。
- 单元测试覆盖加密、文件 schema、Free 限制、重置关键路径。
- 手动验收清单通过，尤其是明文泄露检查和双设备 iCloud 同步检查。
